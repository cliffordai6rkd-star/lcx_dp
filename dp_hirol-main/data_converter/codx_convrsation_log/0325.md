  可行性判断
  把存储格式换成 LeRobot v3 是可行的。官方当前文档明确说 v3 是 file-based 组织：
  data/*.parquet、videos/*.mp4、meta/episodes/*.parquet，目标就是减少文件系统压
  力、提升大规模数据集初始化和存储效率。参考官方文档：

  - https://huggingface.co/docs/lerobot/en/lerobot-dataset-v3
  - https://huggingface.co/docs/lerobot/porting_datasets_v3

  但要讲清一个现实问题：LeRobot v3 能明显改善“写入组织、文件数量、存储开销、初始化
  元数据加载”，不保证你的训练吞吐一定比 zarr 更快。因为 v3 常见路径是图像进 MP4，
  训练时会多一层视频解码；如果你的瓶颈从前是 zarr 写盘和海量小数组，那 v3 很可能更
  好；如果你的瓶颈主要是训练时随机采样多相机历史帧，v3 可能把瓶颈转成 CPU 解码。这
  个需要后面做基准，不应先假设“v3 一定训练更快”。

  我建议的改造路线
  推荐方案不是改 ReplayBuffer，而是新增一条并行数据链路：

  1. 在 data_converter/ 新增“原生 HiROL -> LeRobot v3”转换脚本。
  2. 在 diffusion_policy/dataset/ 新增一个 HirolLeRobotV3Dataset。
  3. 这个新 dataset 直接读 LeRobot v3，但对外仍返回和当前 HirolDataset 一样的
     batch 结构。
  4. 训练配置只切换 task.dataset._target_，workspace、policy、训练循环尽量不动。

  原因很直接：

  - 如果你强行把 LeRobot v3 再伪装成 ReplayBuffer，你会把 v3 的文件级组织优势又重
    新拉回“全量数组视图”，收益会被抵消。
  - 如果你让新 dataset 直接产出当前 batch 契约，训练层几乎不用碰。

  这个新 dataset 的职责应当是：

  - 读取 LeRobot v3 的帧级样本
  - 基于 meta.episodes 做 episode-aware train/val split
  - 自己维护窗口索引和 pad_before/pad_after 逻辑
  - 把 LeRobot 字段名映射成当前 DP 期望的键名
  - 做图像 resize / CHW / float32 / [0,1]
  - 计算 normalizer，接口保持和 base_dataset.py 一致

  字段映射建议
  建议 LeRobot v3 里用更规范的命名做 canonical schema，例如：

  - observation.images.ee_cam_color
  - observation.images.third_person_cam_color
  - observation.images.side_cam_color
  - observation.state
  - action
  - timestamp
  - episode_index
  - 可选：task, subtask, is_terminal, 各类原始 timestamp / valid flag

  然后在 HirolLeRobotV3Dataset 内部映射成当前训练用键：

  - observation.images.ee_cam_color -> ee_cam_color
  - observation.images.third_person_cam_color -> third_person_cam_color
  - observation.images.side_cam_color -> side_cam_color
  - observation.state -> state
  - action -> action

  这样 LeRobot v3 是你的“规范数据层”，Diffusion Policy 侧只是一个 adapter，不会把

• LeRobot v3 里的 feature schema，本质上就是“数据契约”。它定义的不是训练算法，而是
  这份数据里“有哪些字段、每个字段是什么模态、shape 是什么、dtype 是什么、字段之间
  怎么命名”。

  官方文档里，v3 的 canonical schema 记录在 meta/info.json，并和 meta/stats.json、
  meta/episodes/* 一起驱动数据加载、归一化、视频定位和 episode 重建。参考：

  - https://huggingface.co/docs/lerobot/en/lerobot-dataset-v3
  - https://huggingface.co/docs/lerobot/porting_datasets_v3
  - https://huggingface.co/docs/lerobot/introduction_processors

  它具体长什么样
  一个 feature 通常像这样定义：

  {
    "observation.images.wrist_left": {"dtype": "image"},
    "observation.state": {"dtype": "float32", "shape": (8,)},
    "action": {"dtype": "float32", "shape": (8,)},
    "timestamp": {"dtype": "float32", "shape": (1,)},
    "is_first": {"dtype": "bool", "shape": (1,)},
    "is_last": {"dtype": "bool", "shape": (1,)},
    "is_terminal": {"dtype": "bool", "shape": (1,)},
  }

  这些字段名不是随便起的，它们通常带有语义层级：

  - observation.images.*: 图像观测
  - observation.state: 低维观测
  - action 或 action.*: 动作
  - timestamp: 帧时间
  - task_index / subtask_index: 任务标签
  - is_first/is_last/is_terminal: episode 边界和终止标记

  这里最重要的 3 个维度是：

  - name: 字段叫什么，决定下游代码怎么取它
  - dtype: 是 image、float32、bool、string 等
  - shape: 张量维度，比如 (8,)、(7,)、(1,)

  它会影响哪些事情
  它会直接影响下面这些层：

  1. 数据转换脚本
     你的 converter 不是单纯“把文件搬过去”，而是要决定原生字段如何映射到 schema。
     比如你现在是：

  - 三路相机
  - state 或 state_ee
  - action
  - 各种 timestamp / valid flag

  你要决定它们在 LeRobot v3 里是：

  - observation.images.ee_cam_color
  - observation.images.third_person_cam_color
  - observation.images.side_cam_color
  - observation.state
  - action

  还是更细：

  - observation.state.ee_pose
  - observation.state.joint_position
  - action.joint_position
  - action.gripper_width

  这一步一旦定了，后面 adapter 和训练代码都跟着它走。

  2. 存储方式
     v3 不是按 episode 一个文件，而是：

  - 低维数据进 data/*.parquet
  - 图像通常进 videos/*.mp4
  - episode 边界、offset、长度进 meta/episodes/*.parquet

  所以 schema 会决定：

  - 哪些字段写 parquet
  - 哪些字段写 mp4
  - 哪些字段参与 metadata
  - 哪些字段参与 stats

  3. 加载 API 的返回结构
     官方 LeRobotDataset 按 schema 返回字典，比如：

  - sample["observation.state"]
  - sample["action"]
  - sample["observation.images.front_left"]

  这意味着如果你在 schema 里用的是 observation.state，那你未来 adapter 就要从这个
  key 取；如果你拆成 observation.state.joint_position 和
  observation.state.ee_pose，adapter 就要自己再拼接。

  4. 归一化与统计
     meta/stats.json 是按 feature 算统计量的。
     所以 schema 会决定：

  - 哪些字段有 mean/std/min/max
  - 归一化是按整体 observation.state 做，还是按子字段分别做
  - 以后模型输入是不是方便复用现成 normalizer

  对你这个项目尤其重要。你当前 HirolDataset.get_normalizer() 是按
  replay_buffer["action"] 和每个 lowdim key 做的，见 hirol_dataset.py:325。如果未
  来 schema 更细，normalizer 逻辑也会变。

  5. 时序采样方式
     LeRobot v3 原生支持 delta_timestamps 这种按秒取窗口的方式。官方例子里，某个图
     像 key 可以一次返回 [T, C, H, W]。
     这会影响：

  - 你是继续沿用现在 DP 的“按 step index + pad_before/pad_after”采样
  - 还是切换成“按时间戳相对当前帧取历史帧”

  这两种思路都可以，但 schema 里有没有 timestamp、相机帧率是否稳定，会直接影响实现
  难度。

  6. 模型接口与配置
     你当前训练配置吃的是：

  - 三路 RGB
  - 一个 state
  - 一个 action

  见 hirol_fr3_abs_jps_ee_state.yaml:4。

  如果 LeRobot schema 也保留这个扁平结构：

  - observation.images.ee_cam_color
  - observation.images.third_person_cam_color
  - observation.images.side_cam_color
  - observation.state
  - action

  那 adapter 很简单。

  如果你把 schema 设计成很细的层级字段：

  - observation.state.ee_pose
  - observation.state.joint_position
  - observation.state.gripper_width
  - action.joint_position
  - action.gripper_width

  那 adapter 需要在 __getitem__ 里重新拼接出当前模型要的 state 和 action。

  7. 未来扩展性
     schema 设计也决定你以后能不能方便支持：

  - 多机器人
  - 多任务
  - 子任务标注
  - 不同 action space
  - 更换 policy
  - 数据可视化/Hub 展示

  如果你现在只为了接入当前 DP 训练，直接存一个扁平 observation.state 和 action 最
  省事。
  如果你想让这份数据以后也能被 LeRobot 原生训练栈、标注工具、Hub 浏览更自然地使
  用，拆分成更语义化的字段会更好。

  对你当前项目，最关键的影响
  对你来说，feature schema 会先影响 3 件事：

  1. converter 怎么写
     是直接输出：

  - observation.state
  - action
  - 三路图像

  还是输出更细粒度字段。

  2. 新 dataset adapter 怎么写
     如果 schema 和当前 shape_meta 接近，adapter 只做重命名和窗口采样。
     如果 schema 更细，adapter 要做字段拼接、reshape、归一化对齐。
  3. 训练配置要不要改
     如果 adapter 最终仍返回当前格式：

  {
    "obs": {
      "ee_cam_color": ...,
      "third_person_cam_color": ...,
      "side_cam_color": ...,
      "state": ...,
    },
    "action": ...
  }

  那 workspace 和 policy 几乎不用动。

  我对你这边的建议
  先不要急着定“最标准”的大而全 schema。先在两种方案里选一个：

  方案 A，最小改动：

  - LeRobot v3 内部 schema 尽量贴近当前训练需求
  - 核心字段只保留 observation.state、action、三路图像、timestamp、episode 标记
  - 优点：adapter 最简单，最快落地
  - 缺点：未来语义扩展一般

  方案 B，语义更强：

  - 拆成 observation.state.ee_pose、observation.state.joint_position、
    observation.state.gripper_width
  - 动作也拆
  - 额外保留 task_index/subtask_index/valid flag
  - 优点：更 LeRobot 风格，未来更灵活
  - 缺点：你当前 DP 接入层要多写一层拼接逻辑

  如果你愿意，下一步我建议只讨论这个问题：
  “你的 HiROL 数据在 LeRobot v3 里，究竟采用方案 A 还是方案 B，以及每个 key 具体叫
  什么。”
  这一步定完，后面的 converter 和 dataset 结构就基本顺了。
